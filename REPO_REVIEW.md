# Обзор репозитория TowerTech Solutions

## Что находится в репозитории

Репозиторий минималистичный и содержит статический лендинг:

- `index.html` — монолитная страница (HTML + встроенные CSS и JavaScript).
- `vercel.json` — конфигурация деплоя на Vercel как статического сайта.

## Архитектура страницы

Страница собрана по секциям и ориентирована на маркетинговую презентацию услуг:

- Hero (`#top`) с CTA и метриками.
- About (`#about`) с блоком KPI.
- Services (`#services`) — карточки услуг.
- Why Us (`#why-us`) — причины выбора компании.
- Clients (`#clients`) — логотипы/бренды клиентов.
- Safety (`#safety`) — требования/процессы безопасности.
- CTA-секция перед контактами.
- Contact (`#contact`) — контактные данные и форма/призывы.
- Footer.

## Frontend-реализация

### 1) Стили

Весь CSS встроен в `<style>` внутри `index.html`:

- Используются CSS-переменные (`:root`) для палитры и типографики.
- Много кастомных анимаций (`@keyframes`) для glow, floating, blinking и т.п.
- Продумана адаптивность через media queries (мобильная навигация, перестройка grid-блоков).

### 2) Разметка

- Навигация фиксированная, с desktop и mobile версиями.
- Секции имеют якоря (`id`) для smooth-scroll навигации.
- Применяется семантически корректный каркас (`header/main/section/footer`).

### 3) Скрипты

Встроенный JS внизу файла отвечает за интерактив:

- Работа мобильного меню (`hamburger`, открытие/закрытие `mobile-nav`).
- Фоновая графика через `<canvas id="bg-canvas">`.
- Анимационные/UX-поведения (плавные эффекты и мелкие интерактивные элементы).

## Deploy/Infra

`vercel.json` указывает, что:

- Сборка не нужна в классическом смысле — отдается `index.html` как static build.
- Роутинг настроен на отдачу запрошенных путей из статики (простая маршрутизация через `routes`).

## Сильные стороны текущего решения

- Очень быстрый старт и простой деплой.
- Отсутствие runtime-зависимостей.
- Визуально насыщенный UI при нулевой сложности сборки.

## Технические ограничения

- Монолит в одном `index.html` усложняет сопровождение и командную разработку.
- Нет модульности (компонентов, отдельного CSS/JS).
- Нет автоматизированных тестов/линтинга.
- Ограниченная масштабируемость при росте контента.

## Рекомендации по эволюции

1. Вынести CSS и JS в отдельные файлы (`styles/`, `scripts/`).
2. Разделить страницу на компоненты/partials (даже в рамках статического подхода).
3. Добавить базовый quality-gate:
   - HTML-валидация,
   - stylelint/eslint (если появится tooling),
   - Lighthouse CI для перформанса и accessibility.
4. При развитии проекта рассмотреть переход на легковесный SSG/фреймворк (например, Astro/Next static export).
